Key Decisions & Constraints
  • 1
    Audience: Studio owners, not developers Write for a fitness professional setting up their first booking software. Avoid SQL/API jargon in feature docs — save technical depth for the Reference tab only.
    Design
  • 2
    No OpenAPI spec yet — use MDX pages for API reference The v2 app uses Next.js API routes + Supabase RPC — not a REST API with a spec file. The Reference tab documents the database schema and tier gates instead.
    Scope
  • 3
    Tier gates surface throughout feature docs Each feature page will use a <Note> callout where applicable: e.g., "Recurring sessions require the Growth plan or above." Sourced from migration 019.
    UX
  • 4
    Logo assets to be provided The plan assumes Book2befit light/dark SVG logos will replace the current Mintlify placeholders in /logo/. If not ready, the text name is used as fallback.
    Asset
